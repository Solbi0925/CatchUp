# CatchUp MVP 구현 계획

## 1. 목적

CatchUp은 먼저 가짜 데이터(Mock data)로 핵심 사용자 흐름을 끝까지 확인한 뒤, 실제 로그인, 데이터 저장, Google Calendar, AI 기능을 단계적으로 연결한다.

이 문서는 팀이 지금 무엇을 만들고 있는지, 다음 단계로 넘어가도 되는 기준이 무엇인지 함께 확인하기 위한 작업 계획이다. 계획은 구현 과정에서 발견한 내용에 따라 수정할 수 있다.

## 2. 핵심 원칙

- 앱 전체의 방향은 `PRD.md`, `USER_FLOW.md`, `SCREEN_SPEC.md`를 기준으로 한다.
- 화면 하나를 완벽하게 다듬기 전에, `Upload -> AI 추출 -> AI Mate 계획 생성 -> Today/Month 확인`의 핵심 흐름을 Mock 데이터로 먼저 연결한다.
- 실제 개인정보, 실제 학생 자료, API 키, Google OAuth 토큰은 문서나 GitHub에 올리지 않는다.
- 주간 계획 생성은 주 1회, AI Mate 계획 조정은 하루 10회라는 MVP 제한을 처음부터 화면과 데이터에 반영한다.

## 3. 전체 구현 순서

```text
1단계. 공통 설계와 Mock 데이터 준비
-> 2단계. Mock 핵심 흐름 연결
-> 3단계. 화면과 상태 검증
-> 4단계. 실제 데이터 구조와 서버 요청 설계
-> 5단계. 로그인과 데이터 저장 연결
-> 6단계. 업로드, AI, Google Calendar 실제 기능 연결
-> 7단계. 전체 데모와 사용자 테스트
```

---

## 1단계. 공통 설계와 Mock 데이터 준비

### 목표

모든 팀원이 같은 화면 규칙과 같은 가짜 데이터 형식을 기준으로 다음 구현을 시작할 수 있게 한다.

### 담당 작업

| 담당자 | 작업 | 산출물 |
| --- | --- | --- |
| 김세희 | 앱 전체의 공통 디자인 규칙과 화면별 디자인 정리 | `DESIGN_SYSTEM.md`, `DESIGN.md` |
| 박솔비 | 앱에서 저장하고 사용할 핵심 정보 구조 정리 | `DATA_MODEL.md` |
| 심여진 | 화면 확인에 사용할 익명 가짜 사용자, 일정, 과제, AI 결과 작성 | `MOCK_DATA.md` |

### 각 문서의 역할

- `DESIGN_SYSTEM.md`: 색상, 글자 크기, 버튼, 입력창, 여백처럼 앱 전체에 공통으로 적용할 디자인 규칙
- `DESIGN.md`: Today, Month, Upload, AI Mate 등 각 화면에 무엇을 어떤 순서로 배치할지 설명하는 문서
- `DATA_MODEL.md`: 사용자, 업로드 자료, 추출 일정, 개인 일정, 주간 계획, 할 일 등 앱이 다룰 정보의 목록과 관계
- `MOCK_DATA.md`: 실제 서버나 Google Calendar 없이 화면을 확인하기 위한 가짜 일정, 과제, AI 추출 결과, 주간 계획 예시

### 완료 기준

- 네 문서가 `PRD.md`, `USER_FLOW.md`, `SCREEN_SPEC.md`와 충돌하지 않는다.
- `MOCK_DATA.md`의 데이터가 `DATA_MODEL.md`의 항목 이름과 형식을 따른다.
- 가짜 자료는 개인정보가 없는 샘플만 사용한다.
- 팀원 전체가 문서를 함께 읽고, 수정할 부분을 한 번 합의한다.

## 2단계. Mock 핵심 흐름 연결

### 목표

실제 서버, AI, Google Calendar 없이도 CatchUp의 가장 중요한 경험을 직접 눌러볼 수 있게 만든다.

### 구현 순서

1. 온보딩과 Google Calendar 연결 안내 화면을 Mock 상태로 구현한다.
2. Upload 화면에서 샘플 자료를 업로드한 것처럼 보이게 만든다.
3. AI가 과제, 시험, 마감일을 추출한 것처럼 결과를 보여주고 사용자가 수정할 수 있게 만든다.
4. AI Mate 패널에서 가짜 개인 일정과 학업 일정을 바탕으로 주간 계획을 만든 것처럼 보여준다.
5. Today에서 날짜별 할 일과 오늘의 일정을 확인하게 만든다.
6. Month에서 학업 일정과 가짜 개인 일정을 확인하고, 개인 일정을 직접 추가한 것처럼 보이게 만든다.
7. 하단 탭과 모든 화면의 AI Mate 플로팅 버튼을 연결한다.

### 이 단계에서 확인할 제한 규칙

- 주간 계획은 같은 주에 두 번 생성할 수 없는 상태를 보여준다.
- AI Mate 계획 조정은 오늘 남은 횟수를 보여주고, 10회를 모두 사용한 상태도 보여준다.
- 사용자는 Today의 계획 항목 내용을 직접 수정하지 않고 AI Mate로 조정을 요청한다.

### 완료 기준

아래 흐름을 팀원이 직접 눌러서 막힘 없이 확인할 수 있다.

```text
Upload
-> AI 추출 결과 확인 및 수정
-> AI Mate에서 주간 계획 생성
-> Today에서 오늘 할 일 확인
-> Month에서 일정 확인
```

## 3단계. 화면과 상태 검증

### 목표

핵심 흐름이 사용자에게 자연스러운지, 웹 화면의 주요 너비에서 깨지는 부분은 없는지 확인한다.

### 확인 항목

- 데스크톱과 모바일 웹 주요 너비에서 글자, 버튼, 하단 탭, 플로팅 AI Mate 버튼이 겹치거나 잘리지 않는지
- 빈 상태: 아직 자료, 일정, 주간 계획이 없을 때
- 로딩 상태: 자료 추출, 일정 조회, 계획 생성 중일 때
- 오류 상태: 업로드, 추출, Calendar 연결이 실패했을 때
- 성공 상태: 업로드, 추출 확인, 계획 생성이 완료됐을 때
- 탭 이동과 AI Mate 패널 열기/닫기가 자연스러운지

### 완료 기준

팀원 세 명이 같은 흐름을 직접 확인하고, 수정 의견을 화면의 구체적인 위치와 함께 정리한다. 예: "Today의 할 일 카드 제목을 한 줄 더 작게".

## 4단계. 실제 기능 연결 설계

### 목표

Mock으로 확인한 흐름을 실제 데이터로 바꾸기 전에, 저장할 정보와 서버 요청을 명확히 한다.

### 작업

1. `DATA_MODEL.md`를 실제 구현에 필요한 수준으로 보완한다.
2. `API_SPEC.md`를 작성한다.
3. 로그인, 자료 업로드, AI 추출, 추출 결과 저장, Calendar 일정 조회, CatchUp 개인 일정 직접 추가, 주간 계획 생성, 계획 조정, Today/Month 조회 요청을 정리한다.
4. `AI_PROMPTS.md`를 작성한다. 정보 추출, 4주 일정 참고 계획 생성, 계획 조정, 추천 근거 설명의 규칙을 정리한다.

### 완료 기준

각 화면의 버튼이 어떤 정보를 보내고 어떤 결과를 받는지 팀이 설명할 수 있다.

## 5단계. 로그인과 데이터 저장 연결

### 목표

가짜 데이터 대신 사용자별 데이터를 실제로 저장하고 다시 불러올 수 있게 만든다.

### 구현 순서

1. Auth 연결: 사용자를 구분하는 로그인 기능을 연결한다.
2. DB 연결: 사용자, 업로드 자료, 추출 결과, 일정, 주간 계획, 할 일을 저장한다.
3. Today와 Month가 저장된 데이터를 읽어 표시하도록 연결한다.

### 완료 기준

테스트 사용자가 로그인한 뒤 만든 데이터가 저장되고, 앱을 다시 열어도 같은 사용자 데이터로 확인된다.

## 6단계. 실제 핵심 기능 연결

### 목표

Mock 기능을 실제 업로드, AI, Google Calendar 데이터로 교체한다.

### 구현 순서

1. 파일 업로드와 저장 기능을 연결한다. 최초 MVP에서는 PDF와 이미지 파일 형식을 모두 지원한다.
2. AI 정보 추출 기능을 연결하고, 추출 결과를 사용자가 확인 및 수정한 뒤 저장하게 만든다.
3. Google Calendar 실제 연동을 연결한다.
4. CatchUp 안에서 개인 일정을 직접 추가하고 저장하는 기능을 연결한다.
5. AI가 업로드 자료, 저장된 학업 일정, Google Calendar 개인 일정과 CatchUp에서 직접 추가한 개인 일정, 앞으로 4주 일정을 참고해 이번 주 계획을 만들게 한다.
6. AI Mate 계획 조정과 추천 근거 설명 기능을 연결한다.
7. 주 1회 생성 제한과 하루 10회 조정 제한을 실제 저장 데이터 기준으로 적용한다.

### Google Calendar 진행 방식

Google Calendar는 두 번 나누어 진행한다.

1. **2단계와 병행하는 간단한 기술 확인**
   - Google 로그인 권한 허용(OAuth)이 가능한지 확인한다.
   - 테스트용 Calendar에서 일정 목록을 한 번 읽어올 수 있는지 확인한다.
   - API 키나 OAuth 토큰은 GitHub와 문서에 올리지 않는다.

2. **6단계의 실제 화면 연동**
   - Mock 개인 일정을 실제 Google Calendar 일정으로 교체한다.
   - 가져온 일정이 Today와 Month에 표시되는지 확인한다.
   - AI Mate의 주간 계획 생성에 개인 일정이 반영되는지 확인한다.

### 완료 기준

익명 테스트 일정과 샘플 학업 자료로 Upload, AI Mate, Today, Month가 실제 데이터로 연결된다.

## 7단계. 전체 데모와 사용자 테스트

### 목표

발표 전에 실제 사용자 흐름이 처음부터 끝까지 작동하는지 확인하고, 핵심 문제를 보완한다.

### 최종 데모 흐름

```text
앱 실행
-> Google Calendar 연결
-> 학업 자료 업로드
-> AI 추출 결과 확인 및 수정
-> 주간 계획 생성
-> Today에서 오늘 할 일 확인 및 완료 체크
-> Month에서 일정 확인
-> 필요하면 개인 일정 직접 추가
-> AI Mate로 계획 조정 요청
```

### 최종 확인 항목

- 주간 계획 생성이 한 주에 한 번만 가능한지
- AI Mate 조정 요청이 하루에 열 번까지만 가능한지
- Google Calendar 연결 실패 시 재시도 안내가 보이는지
- 추출 실패 또는 확인 필요 항목을 사용자가 수정할 수 있는지
- 추천 이유가 수치형 완료 확률이 아니라 이해하기 쉬운 근거로 표시되는지
- 테스트 사용자에게 "오늘 무엇을 해야 하는지 빠르게 알 수 있었는지"를 묻고 의견을 기록하는지

## 4. 문서 수정 원칙

- 구현 중 화면 흐름이 바뀌면 먼저 `USER_FLOW.md`와 `SCREEN_SPEC.md`를 수정한다.
- 공통 디자인 규칙이 바뀌면 `DESIGN_SYSTEM.md`를 수정한다.
- 새 데이터가 필요해지면 `DATA_MODEL.md`와 `MOCK_DATA.md`를 함께 수정한다.
- 실제 서버 요청이 바뀌면 `API_SPEC.md`를 수정한다.
- 문서는 한 번 작성하고 끝나는 결과물이 아니라, 구현과 함께 최신 상태로 유지하는 팀의 설계도다.
